### 类型

- 哈希表

	- 时间复杂度O(1)

	- 解决哈希冲突

		- Redis的hash表是全局的，所以当写入大量的key时，将会带来哈希冲突，以及rehash可能带来的操作阻塞

		- Redis解决hash冲突的方式，是链式哈希：同一个哈希桶中的多个元素用一个链表来保存

		- 当哈希冲突链过长时，Redis会对hash表进行rehash操作。rehash就是增加现有的hash桶数量，分散entry元素

	- rehash机制

		- 为了使rehash操作更高效，Redis默认使用了两个全局哈希表：哈希表1和哈希表2，起始时hash2没有分配空间

		- 随着数据增多，Redis执行分三步执行rehash

			- 1，给hash2分配更大的内存空间，如是hash1的两倍

			- 2，把hash1中的数据重新映射并拷贝到哈希表2中

			- 3，释放hash1的空间

		- 渐进式rehash

			- rehash非常耗时，会阻塞redis

			- 迁移数据，改成每处理一个请求时，就从hash1中的第一个索引位置，顺带将这个索引位置上的所有entries拷贝到hash2中

			- 问题

				- 如果旧表还没有完全同步到新表，新表却再次需要rehash了，怎么处理的呢

					- rehash时，进行扩容时，会进行判断，应该是等到同步完到新表之后，再次同步

- 跳表

	- 时间复杂度O(logN)

	- 在链表的基础上增加了多级索引，通过索引的几次跳转，实现数据快速定位

- 双向链表

	- O(N)

- 压缩链表

	- O(N)

	- 压缩列表类似于一个数组，不同的是:压缩列表在表头有三个字段zlbytes,zltail和zllen分别表示长度，列表尾的偏移量和列表中的entry的个数，压缩列表尾部还有一个zlend，表示列表结束，所以压缩列表定位第一个和最后一个是O(1),但其他就是O(n)

- 整数数组

	- O(N)

### 不同操作的复杂度

- 单元素操作

	- 是指每一种集合类型对单个数据实现的增删改查操作

		- O(1)

- 范围操作

	- 是指集合类型中的遍历操作，可以返回集合中的所有数据

		- O(N)

- 统计操作

	- 是指集合类型对集合中所有元素个数的记录

		- O(1)

- 例外情况

	- 是指某些数据结构的特殊记录

		- 例如压缩列表和双向链表都会记录表头和表尾的偏移量

			- 对于 List 类型的 LPOP、RPOP、LPUSH、RPUSH 这四个操作来说，它们是在列表的头尾增删元素，这就可以通过偏移量直接定位，所以它们的复杂度也只有 O(1)，可以实现快速操作

## IO模型

### 单线程模型

- Redis 的网络 IO 和键值对读写是由一个线程来完成的，这也是 Redis 对外提供键值存储服务的主要流程

- 基于多路复用的高性能 I/O 模型

- 性能，qps达到10w/s

- 问题

	- 为啥不采用多线程

		- 多线程需要对共享资源并发进行控制，比如同时修改一个列表，如果只是增加一个互斥锁，那么并行变串行，系统吞吐量不会随着线程的增加而增加

	- 单线程redis为啥那么快

		- 主要操作在内存中完成，且使用了高效的数据结构，如hash表、跳表等

		- 采用多路复用机制，实现高吞吐量

## 持久化

### AOF（Append Only File）日志

- 定义

	- 写后日志，即先执行命令，把数据写入内存，然后才记录日志（主线程操作）

		- 好处

			- 避免了检查命令语法是否正确的开销，因此只有命令成功执行，才会记录

			- 在命令执行后才记录日志，所以不会阻塞当前的写操作（即先写内存，再写日志）

- 写回策略（配置项appendfsync）

	- Always（同步写回）

		- 每个写命令执行完，立马同步地将日志写回磁盘

	- Everysec（每秒写回）

		- 每个写命令执行完，只是先把日志写到 AOF 文件的内存缓冲区，每隔一秒把缓冲区中的内容写入磁盘

	- No（操作系统控制的写回）

		- 每个写命令执行完，只是先把日志写到 AOF 文件的内存缓冲区，由操作系统决定何时将缓冲区内容写回磁盘

- 日志文件太大了怎么办

	- AOF 重写机制（子进程）

		- 触发条件

			- 手动触发

				- 用户通过调用BGREWRITEAOF

			- 自动触发

				- 没有BGSAVE命令（RDB持久化）/AOF持久化在执行

				- 没有BGREWRITEAOF在进行

				- 当前AOF文件大小要大于server.aof_rewrite_min_size（默认为1MB），或者在redis.conf配置了auto-aof-rewrite-min-size大小

				- 当前AOF文件大小和最后一次重写后的大小之间的比率等于或者等于指定的增长百分比（在配置文件设置了auto-aof-rewrite-percentage参数，不设置默认为100%，当前AOF文件大小和上一次重写后AOF文件大小的差值，再除以上一次重写后AOF文件大小）

		- 根据键值对最新状态，生成写入命令（多条命令=>1条命令）

		- 过程

			- 1. fork子进程

			- 2. 子进程开始将新AOF文件的内容写入到临时文件

			- 3. 对所有新执行的写命令，父进程一边将它们追加到日志缓冲区，一边追加到原有的AOF文件中（若重写AOF失败不影响该AOF文件）

			- 4. 当子进程完成AOF重写后，会向父进程发送信号，父进程收到信号后，会将缓冲区的日志追加到新AOF文件末尾

			- 5. 对新AOF文件进行改名，原子覆盖原有的AOF文件

- 问题

	- 在AOF重写的过程中，如果有新的写命令到来了，会影响AOF重写吗？

		- 当然不会，新的写命令不仅会记录到AOF日志的缓存区，还会记录到重写的新AOF日志缓存区，这样当AOF重写结束后，把重写缓存区数据写到新AOF文件，就不会丢失了

	- AOF 重写过程中有没有其他潜在的阻塞风险？

		- 主线程fork子进程时，子进程要copy父进程的页表，如果父进程的内存比较大，那么页表就会大，也就越耗时（只是页表，不是内存copy）

			- [node:](https://draveness.me/whys-the-design-linux-default-page/)

				- 页表就是一个存储物理页地址的表，存储虚拟地址和物理地址的映射关系，一般一个页表为4k

		- 子进程和主进程共享内存，当主进程收到写操作时，主进程会新申请空间进行保存新写的数据，如果这个key-value数据比较大，耗时就比较长

		- 如果命令执行成功，还没来得及写入AOF日志，服务宕机了，这条数据就会丢失

		- 虽然AOF避免了对档条写日志的阻塞，当时AOF依然在主线中完成，如果磁盘满了造成了的阻塞，依然会造成对下一个命令的阻塞

### RDB (redis database)快照

- 命令

	- save命令

		- 在主线程中执行，会导致阻塞

	- bgsave命令

		- 创建一个子进程，专门用于写入 RDB 文件，避免了主线程的阻塞（默认配置）

- 写时复制技术（Copy-On-Write, COW）

	- COW核心思路

		- fork一个子进程，只有在父进程发生写操作修改内存数据时，才会真正去分配内存空间，并复制内存数据，而且也只是复制被修改的内存页中的数据，并不是全部内存数据，子进程持久化的RDB文件依然是老数据

		- 内核会将主进程的所有内存页权限设置为read-only权限，然后子进程的地址空间指向主进程，即共享了主进程的内存，当主进程修改内存时，cpu硬件检测到内存页是read-only权限，于是触发也异常中断page-fault），在中断例程中，内核就会把发生异常的页复制一份，而子进程仍然使用的是老数据

	- 保证在执行快照的同时，正常处理写操作

	- 总体来看，Redis还是读操作比较多。如果子进程存在期间，发生了大量的写操作，那可能就会出现很多的分页错误(页异常中断page-fault)，这样就得耗费不少性能在复制上。

	- 在rehash阶段上，写操作是无法避免的。所以Redis在fork出子进程之后，将负载因子阈值提高，尽量减少写操作，避免不必要的内存写入操作，最大限度地节约内存

- 快照频率

	- 频繁快照缺点

		- 频繁将全量数据写入磁盘，会给磁盘带来很大压力，多个快照竞争有限的磁盘带宽

		- bgsave 子进程需要通过 fork 操作从主线程创建出来，fork 这个创建过程本身会阻塞主线程，而且主线程的内存越大，阻塞时间越长

	- 混合使用 AOF 日志和内存快照

		- 内存快照以一定的频率执行

		- 在两次快照之间，使用 AOF 日志记录这期间的所有命令操作

		- 等到下次做全量快照时，就可以清空 AOF 日志

## 高可靠集群

### 主从模式

- 数据同步一致性

	- 读写分离

		- 操作

			- 读操作

				- 主库、从库都可以接收

			- 写操作

				- 首先到主库执行，然后，主库将写操作同步给从库

		- 好处

			- 所有数据的修改只会在主库上进行，不用协调三个实例（加锁等），减小开销

	- 主从库数据同步过程

		- 流程图

			-  

				- sync.jpg

		- 步骤

			- 第一阶段

				- 从库发送psync 命令进行数据同步，包含了主库的 runID 和复制进度 offset 两个参数

					- runID

						- 每个 Redis 实例启动时都会自动生成的一个随机 ID，用来唯一标记这个实例，当从库和主库第一次复制时，因为不知道主库的 runID，所以将 runID 设为“?”

					- offset

						- -1，表示第一次复制

				- 主库收到 psync 命令后，会用 FULLRESYNC 响应命令带上两个参数：主库 runID 和主库目前的复制进度 offset，返回给从库

			- 第二阶段

				- 主库执行 bgsave 命令，生成 RDB 文件，接着将文件发给从库

				- 从库接收到 RDB 文件后，会先清空当前数据库，然后加载 RDB 文件

			- 第三阶段

				- 当主库完成 RDB 文件发送后，就会把此时 replication buffer （保存了同步过程中收到的新的写命令）中的修改操作发给从库，从库再重新执行这些操作

	- 减小主从同步过程中主库压力

		- 主从级联模式（“主 - 从 - 从”模式）

			- 部署主从集群的时候，可以手动选择一个从库（比如选择内存资源配置较高的从库），用于级联其他的从库

			- 可以选择一些从库（例如三分之一的从库），在这些从库上执行如下命令，让它们和刚才所选的从库，建立起主从关系

				- replicaof  所选从库的IP 6379

- 问题

	- 主从库网络断开情况

		- 架构图

			-  

				- redis-replication.jpg

		- 只要从库存在，那么主库会将所有写命令，处理传播给从库外，还会写入repl_backlog_buffer缓冲区，而当从库发送同步请求后，主库会将增量命令从repl_backlog_buffer缓冲区发送到replication buffer，进而通过网络发送给从库

			- repl_backlog_buffer （在主库里申请，所有从库共享，支持从库增量同步）

				- 定义

					- 是一个环形缓冲区，主库会记录自己写到的位置，从库则会记录自己已经读到的位置

				- 大小计算

					- 缓冲空间大小 = 主库写入命令速度 * 操作大小 - 主从库间网络传输命令速度 * 操作大小

					- repl_backlog_size = 缓冲空间大小 * 2

		- 主从库的连接恢复之后

			- 从库首先会给主库发送 psync 命令，并把自己当前的 slave_repl_offset 发给主库

			- 主库会判断自己的 master_repl_offset 和 slave_repl_offset 之间的差距

				- 如果repl_backlog_buffer 环形缓冲区没有覆盖掉之前的写操作

					- 增量同步

				- 如果repl_backlog_buffer 环形缓冲区覆盖了之前的写操作

					- 全量同步

	- 在主从切换过程中，客户端能否正常地进行请求操作呢

		- 主从模式一般采用读写分离，如果主库故障后，只会影响写操作，读操作不影响

	- 数据库主从不一致问题解决方案

		- 业务是否可以接受短时间不一致，可以则无需优化

		- 强制读主，高可用主库，用缓存提高读性能

		- 在cache里记录哪些记录发生过写请求，并且记录key设置超时时间，如果key过期了，路由到从，否则路由主

	- 主从同步为啥使用RDB文件而不是AOF文件

		- 1. RDB是内存二进制文件，不同的数据类型做了针对性优化，文件比较小， 而AOF文件是记录的每一次操作命令，可能对同一个key又多次冗余操作（AOF文件重写前），文件比较大，而且从库再收到AOF文件后，还需要进行命令重放，过程耗时

		- 2. 如果使用AOF文件，需要开启AOF功能，这样对性能会有所影响

- 优缺点

	- 优点

		- 主从复制，提高可用性

		- 从服务可以提供读服务，分摊主服务读压力

	- 缺点

		- 不具备自动容错和恢复能力

		- 较难支持在线扩容，在集群容量达到上限时(数据量大，可能导致各种慢问题)，在线扩容会变得复杂

### 哨兵模式

- 哨兵作用

	- 监控

		- 作用

			- 周期性地给所有的主从库发送 PING 命令，检测它们是否仍然在线运行

		- 类型

			- 主库

				- 通过哨兵集群，大于一半的哨兵认为”主观下线“，标记为”客观下线“

			- 从库

				- PING 命令的响应超时了，标记为“主观下线”

	- 选主

		- 哨兵本身选主

			- 1. 当哨兵自身判断主库主观下线后，会给其他哨兵发送is-master-down-by-addr命令

			- 2. 其他哨兵会根据自身和主库的连接情况，做出Y或N回应

			- 3. 当一个哨兵获取到赞成票数>=配置中的quorum，就可以标记主库客观下线

			- 4. 当主库被哨兵标记为客观下线后，该哨兵就可以发起哨兵leader选举，表明希望自己来执行主从切换

			- 5. leader哨兵满足的条件

				- 拿到半数以上的赞成票

				- 拿到的票数同时还需要大于等于哨兵配置文件中的quorum

		- 选主库

			- 筛选

				- 检查从库的当前在线状态

				- 判断它之前的网络连接状态

					- 配置项 down-after-milliseconds * 10

						- 如果在 down-after-milliseconds 毫秒内，主从节点都没有通过网络联系上，我们就可以认为主从节点断连了。如果发生断连的次数超过了 10 次，就说明这个从库的网络状况不好，不适合作为新主库

			- 打分

				- 第一轮

					- 优先级最高的从库得分高

						- slave-priority 配置项

				- 第二轮

					- 和旧主库同步程度最接近的从库得分高

						- slave_repl_offset最接近master_repl_offset

				- 第三轮

					- ID 号小的从库得分高

	- 通知

		- 1. 把新主库的连接信息发给其他从库，让它们执行 replicaof 命令，和新主库建立连接，并进行数据复制。

		- 2. 哨兵会把新主库的连接信息通知给客户端，让它们把请求操作发到新主库上

- 优缺点

	- 优点

		- 主从可以自动切换，可用性更高

	- 缺点

		- 较难支持在线扩容，在集群容量达到上限时(数据量大，可能导致各种慢问题)，在线扩容会变得复杂

### 集群模式（一般使用主从复制集群）

- 分片存储

	- 一共有16384 个哈希槽，每个key通过 CRC16 校验后对16384 取模来决定放置哪个槽

- 优缺点

	- 优点

		- 方便在线扩缩容，添加、删除节点较方便

## redis6.0新特性

### 多IO线程

- 之前主要依靠单线程io多路复用机制，一个线程监听多个socket读写时间

- 6.0采用多线程（线程池）并行读取网络、协议转换以及数据写回socket，其中每个线程仍旧采用io多路复用机制，注意多线程并不处理命令，命令仍旧有主线程完成执行

- 使用与提升吞吐量场景

### 客户端缓存

- 普通模式

	- 服务端监控客户端读取过的key，如果key的值发生变化，服务端就会发送invalidate消息通知客户端，

	- 注意如果通知客户端无效后，客户端没有再次读取该key，那么即使后续该key的值再次被修改，也不会再次通知客户端（设计的原则：减少服务端监控key的成本，如果客户端不再进行读取该key，服务端也就无需监控该key）

- 广播模式

	- 先注册需要监控的key，后续如果该key的值发生了变化，会以广播的形式通知到客户端

	- 注意：不同于普通模式，如果客户端没有读取该key，后续该key的值再次被修改，仍会通知客户端

### 访问权限控制

- 区分不同过用户，不同用户密码分别不同

	- 每个用户可以设置命令操作权限

	- 针对key粒度控制

	- 针对命令粒度控制

### RESP3

- 数据编码-直接通过不同的开头字符，区分不同的数据类型
